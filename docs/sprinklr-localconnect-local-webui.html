Developer runbook

Run QA6 WebUI requests locally

Bring up the WebUI on your laptop, establish the QA6 tunnel, opt only your browser into routing, and prove where every request is going.

Before you start

QA6 must be ready

  • Coordinator, router and relay pods are Ready.
  • The three Services have endpoints.
  • The WebUI canary Ingress exists for the target hostname and path.
  • QA6 Redis is reachable by the pods.

Laptop must be ready

  • Java 21 or newer.
  • ssh, curl, nc and openssl.
  • Access to the QA6 jump host and Kubernetes host.
  • A local WebUI listening on port 8080.
kubectl -n spr-apps get pods,svc,endpoints | grep spr-local-connect
kubectl -n spr-apps get ingress | grep spr-local-connect
Fresh-clone note: the launcher invokes ./gradlew. The wrapper JAR is intentionally ignored in this repository, so a fresh clone needs a working Gradle installation or an approved copy of gradle-wrapper.jar. Your existing checkout already contains it.

1. Start the local WebUI

LocalConnect does not start WebUI. It forwards traffic only after your application is independently running and healthy.

Configure Java and Tomcat

Use Java 21 and Tomcat 10.1.x. In IntelliJ, deploy the local exploded WebUI artifact with context path /ui and HTTP port 8080.

Load the normal local-development configuration

Keep the WebUI's required local vault, Redis/cache, Spring placeholder and environment properties. LocalConnect does not replace application configuration.

Wait for the WebUI context to finish

Tomcat listening is insufficient if the /ui context failed to initialize. Confirm there is no context startup failure in the server log.

Probe the application directly

Verify the same path and port that the tunnel client will call.

curl -I http://127.0.0.1:8080/ui/ping.html
lsof -nP -iTCP:8080 -sTCP:LISTEN
Ready means both conditions hold: port 8080 is listening and the /ui application context answers. A Tomcat process with a failed WebUI deployment is not ready.

Local application errors are separate from tunnel errors

Local logMeaning
cacheManager ... 'beanName' must not be emptyA required Spring placeholder/property is missing; fix the local WebUI configuration before tunnelling.
GraphQLSchemaGenerator ... dynamic proxyThe request reached local Tomcat, but local GraphQL schema initialization failed. This proves routing can work; fix the local application.
/proc/1/fd/1 ... No such fileA container-oriented Logback appender is being used on macOS. Override local logging so diagnostics remain visible.

2. Start the LocalConnect tunnel

cd /Users/lakshay.jindal/Desktop/sprinklr_repos/spr-local-connect
scripts/qa6-local-connect.sh 8080

The script will prompt for the QA6 Kubernetes host password and then sudo. Keep this terminal open.

Validate and build the client

The script checks port 8080, builds client-0.1.0-SNAPSHOT.jar, and generates a random session ID.

Open the outer SSH path

Laptop → access.qa6.spr-ops.com → 10.56.32.106, then sudo su - sprapps.

Expose QA6 services

Remote kubectl port-forward processes expose the coordinator and relay, while SSH maps them back to laptop ports 8090 and 2222.

Login and assign a relay

The client validates or refreshes its token, asks the coordinator for a relay, and creates a session record in Redis.

Open the inner SSH tunnel

The client connects to the assigned relay through localhost:2222 and registers the session.

Start the inspector

Only after the client starts, it listens on localhost:4040 and forwards tunnel traffic to 127.0.0.1:8080.

Required success messages

Ready: QA6 coordinator on localhost:8090
Ready: QA6 relay SSH on localhost:2222
tunnel established
forwarding to 127.0.0.1:8080
inspect requests at http://localhost:4040
If the script stops before these messages, port 4040 will be down. The inspector is part of the Java client; it is not served by Tomcat.

3. Opt your browser into local routing

The script prints a unique session ID but does not modify your browser. Set both cookies manually on the exact QA6 hostname you are using.

Ingress selector

Name:  sprLocalConnect
Value: always
Path:  /ui

This makes the NGINX canary Ingress select the shared LocalConnect router for matching /ui requests.

Developer session

Name:  remoteDebugConf
Value: <session ID printed by script>
Path:  /ui

This tells the router which developer session and relay connection to use.

In Edge or Chrome: open DevTools → Application → Storage → Cookies → select the exact hostname, create/update both rows, then reload.

Delete SPR_STICKINESS for this hostname before testing. Existing WebUI affinity can otherwise keep the request on the normal backend. Also remove duplicate or obsolete remoteDebugConf, ngrokAlt, and old session cookies.
Cookie PathEffect
/ui (recommended)The browser sends the LocalConnect cookies only for paths beginning with /ui. Frontend pages such as /care/... remain on normal QA6.
/The browser sends the cookies for every path on the hostname. Use only when intentionally routing the complete host.
Other users are unaffected. Their browsers lack sprLocalConnect=always, so they continue to the normal QA6 WebUI Service.

Then trigger a browser request under /ui, for example a GraphQL call. Watch three places:

Router log

Should show session → relay → relay HTTP address.

Inspector :4040

Should record the request method, URL, status, duration and byte counts.

Local Tomcat

Should log or hit your breakpoint for the same /ui/... request.

kubectl -n spr-apps logs -f deploy/spr-local-connect-tier1-router
kubectl -n spr-apps logs -f deploy/spr-local-connect-tier1-relay
A local 401 or 500 still proves tunnelling. If the request appears in port 4040 and local Tomcat, routing succeeded; authentication or application startup is then the separate problem.

Redis checks

HGETALL spr-local-connect:qa6-tier1:session:<session-id>
SCAN 0 MATCH spr-local-connect:qa6-tier1:pod:*

SCAN is cursor-based. Continue using the returned cursor until it returns 0; an empty page with a nonzero cursor does not mean no keys exist.

Local port checks

lsof -nP -iTCP:8080 -sTCP:LISTEN
lsof -nP -iTCP:8090 -sTCP:LISTEN
lsof -nP -iTCP:2222 -sTCP:LISTEN
lsof -nP -iTCP:4040 -sTCP:LISTEN

Stopping safely

Press Ctrl+C in the launcher terminal. The script stops the Java client, closes the outer SSH connection, and terminates the remote Kubernetes port-forwards. Ports 4040, 8090, and 2222 then close. Redis session and heartbeat records expire through TTL.

Remove the two browser cookies when you want all traffic to return to normal QA6:

sprLocalConnect
remoteDebugConf
