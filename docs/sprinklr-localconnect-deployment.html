Merge to the default branch to build three images and upgrade the QA6 runtime automatically, then verify the WebUI canary Ingress and the complete control and data planes.

What gets deployed

Coordinator

Login, token issuance, session creation and relay assignment. Image: spr-local-connect-coordinator.

Router

Forwards selected HTTP requests on 8080 and accepts session-aware HTTP CONNECT traffic for gRPC/raw TCP on 8181. Image: spr-local-connect-router.

Relay

Owns the persistent SSH connection and carries HTTP plus raw TCP streams to the laptop; its cluster-internal TCP bridge listens on 8182. Image: spr-local-connect-relay.

One Helm release, three Deployments. Redis and Kafka already exist in QA6. The optional gateway prototype and laptop client are not deployed by this chart.
Default path: merge the MR. A push pipeline on the protected default branch builds coordinator, router and relay through Jenkins, exports their image tags, and deploys qa6-tier1 only after every build succeeds. The GitLab project needs protected, masked QA6_JENKINS_USER and QA6_JENKINS_TOKEN variables. The manual steps below are the recovery and troubleshooting path.

2. Manual runtime-deployment fallback

Normally this runs automatically after all three image builds succeed. For recovery or an explicit rollback, open custom-k8s-helm-deployment and supply the following values.

ParameterValue

CHART_NAMEspr-local-connect
CHART_RELEASE_NAMEqa6-tier1

CHART_REPO_BRANCHorigin/main

EXTRA_VALUES:

--set-string=image.coordinator.tag=<COORDINATOR_TAG>,image.router.tag=<ROUTER_TAG>,image.relay.tag=<RELAY_TAG>
Keep --set-string. Timestamp-shaped image tags otherwise become integers and Helm can render invalid image names such as %!s(int64=...). The QA6 release file already selects asia-south1-docker.pkg.dev/gc-qa6/gc-qa6.
Fallback safety is configured by the QA6 release. It sets FALLBACK_ENVIRONMENT=qa6 and FALLBACK_DOMAIN_SUFFIX=.sprinklr.com. Dynamic replay accepts only matching QA6 hostnames and always uses HTTPS.

3. Deploy the WebUI canary Ingress

The runtime alone cannot receive public WebUI traffic. The WebUI chart adds a static QA6 canary Ingress that selects sprLocalConnect=always and sends matching paths to spr-local-connect-tier1-router:8080.

This Ingress is required only for local WebUI HTTP routing. Downstream microservice mode leaves the HTTP request on normal QA6 WebUI; that WebUI opens a cluster-internal CONNECT connection to router port 8181.

The LocalConnect WebUI chart changes are merged. Deploy the normal origin/master branch; the former feature branch is no longer required.

4. Verify the deployment

kubectl -n spr-apps get deploy,svc | grep spr-local-connect
kubectl -n spr-apps get pods | grep spr-local-connect
kubectl -n spr-apps get endpoints \
  spr-local-connect-tier1-coordinator \
  spr-local-connect-tier1-router \
  spr-local-connect-tier1-relay

kubectl -n spr-apps get ingress | grep spr-local-connect

Expected Deployments and Services:

spr-local-connect-tier1-coordinator
spr-local-connect-tier1-router
spr-local-connect-tier1-relay

Verify the ports explicitly; pod readiness alone does not prove the raw data plane is exposed:

kubectl -n spr-apps get svc spr-local-connect-tier1-router \
  -o jsonpath='{.spec.ports}{"\n"}'
kubectl -n spr-apps get deploy spr-local-connect-tier1-relay \
  -o jsonpath='{.spec.template.spec.containers[0].ports}{"\n"}'
kubectl -n spr-apps logs deploy/spr-local-connect-tier1-relay \
  | grep 'raw TCP bridge listening on 8182'

Expected: router Service ports 8080 and 8181; relay container ports 8080, 2222, and 8182.

Confirm health from inside the cluster:

kubectl -n spr-apps port-forward svc/spr-local-connect-tier1-coordinator 8090:8090
curl http://127.0.0.1:8090/actuator/health

Expected: all three pods Ready, all three Services populated with endpoints, coordinator health UP, and WebUI canary Ingresses present.

Deployment troubleshooting

SymptomCause and action
Build skipped after checkoutJenkins/Git whatchanged incompatibility. Fix changelog generation; changing Docker arguments will not help.
InvalidImageName with %!s(int64=...)Tags were parsed as numbers. Redeploy using --set-string.
Service not found during tunnel startupThe chart is absent, wrong branch was used, or the expected release was not installed.
Service has no endpointsPods are not Ready. Inspect kubectl describe pod and component logs.
Request never reaches routerCanary Ingress is missing for that WebUI release/host/path, selector cookie is absent, or old stickiness is overriding selection.
CONNECT resets immediatelyThe router process may listen on 8181 while its Kubernetes Service exposes only 8080. Redeploy the chart and verify both Service ports.
502 Tunnel Not Found from CONNECTThe session key is absent/stale or the assigned relay does not own the active SSH session. Confirm the session printed at startup matches the established session and exists in Redis.
Router reports no live relay for HTTPSession is old/expired or relay heartbeat is absent. The HTTP router strips LocalConnect routing cookies and replays through the original QA ingress. Start a fresh tunnel to restore local routing.
gRPC works but nothing appears on 4040Expected. CONNECT forwards an untouched raw stream and bypasses the HTTP inspector; use a local breakpoint, application log, or listening-port check.
