High-level architecture

The browser never connects directly to the laptop. The laptop creates an outbound SSH tunnel, and QA6 reuses that connection to deliver selected HTTP requests.

Core components

Coordinator

The control plane on port 8090.

  • Authenticates developers
  • Issues and validates tokens
  • Discovers live relay pods
  • Creates and reassigns sessions
  • Writes assignment state to Redis

Router

The session-aware HTTP router on 8080 and raw-stream CONNECT proxy on 8181.

  • Reads remoteDebugConf
  • Resolves session and pod in Redis
  • Adds tunnel session and trace headers
  • Forwards the complete HTTP request
  • Routes gRPC/TCP connections without decoding them

Relay

Bridges cluster HTTP traffic into the persistent SSH tunnel.

  • Accepts router traffic on 8080
  • Accepts raw router streams on 8182
  • Accepts client SSH on 2222
  • Maps sessions to live connections
  • Multiplexes concurrent requests

Redis

Stores short-lived coordination state.

  • Session → assigned relay pod
  • Pod → host and port
  • Authentication token → owner
  • Session and pod heartbeat TTLs
  • No microservice name is stored

Local client

The developer-side process that owns the reverse tunnel.

  • Requests an assignment
  • Maintains the SSH connection
  • Forwards to localhost:8080
  • Reconnects after interruptions

Request inspector

A lightweight Java HTTP server on 4040, not Tomcat.

  • Shows HTTP requests received by the laptop
  • Records status, duration and byte counts
  • Exposes /api/requests
  • Does not inspect raw TCP/gRPC streams

Tunnel startup

The developer starts a local application and then launches one script. The script creates both the infrastructure access path and the application tunnel.

cd /Users/lakshay.jindal/Desktop/sprinklr_repos/spr-local-connect
scripts/qa6-local-connect.sh 8080
Validate and build

Verify localhost:8080, build the client JAR and generate a unique session ID.

Open outer infrastructure SSH

Connect from the laptop through the QA6 access host to 10.56.32.106.

Start Kubernetes port-forwards

Under the sprapps user, map random remote ports to coordinator 8090 and relay SSH 2222.

Create stable laptop endpoints

SSH -L forwarding exposes the random remote ports as laptop ports 8090 and 2222.

Authenticate and request assignment

The client calls the coordinator on localhost:8090. The coordinator chooses a live relay and writes the session to Redis.

Start inspector and inner SSH tunnel

The client serves the inspector on 4040, connects to localhost:2222, authenticates and registers the session.

Keep the session alive

Client and relay heartbeats refresh Redis TTLs. The coordinator remains reachable for automatic reassignment after an interruption.

Why keep port 8090 open? The active request path does not use it, but the client needs the coordinator to reassign the existing session after a relay restart, pod rollout or tunnel interruption.

Selective microservice/gRPC plane

remoteDebugConf=<session-id>:<exact-tier-alias> is read by the normal QA6 WebUI. Do not set sprLocalConnect in this mode: the WebUI request stays in QA6, while only calls using the matching caller-side tier configuration are intercepted.

The tunnel is generic. Startup registers only the session and assigned relay. The tier alias is consumed inside the QA6 caller and is not sent to or stored by the router, relay, or Redis. Change the cookie suffix to reuse the same live tunnel for another tier. The verified process-engine flow uses ui-process-engine; a URL segment such as process-engine is not necessarily the configured alias.

What is preserved?

The client forwards the original HTTP method, path, query string, headers, cookies and body to the local service. The status, response headers and body return through the same tunnel.

POST https://space-qa6-app-workflow.sprinklr.com/ui/graphql/searchStandardEntities
                                  ↓
POST http://127.0.0.1:8080/ui/graphql/searchStandardEntities
HTTP WebUI mode: remove the existing SPR_STICKINESS cookie when enabling the canary tunnel. It may otherwise keep the browser pinned to the normal WebUI backend. Microservice mode stays on normal QA6 WebUI and does not use the canary selector.

Multi-user model

The infrastructure is shared, while Redis session IDs isolate developer traffic.

Lakshay

remoteDebugConf=c38110...

Routes to Lakshay's registered client connection and laptop.

Developer B

remoteDebugConf=abc123...

Routes to Developer B's registered client connection and laptop.

Developer C

remoteDebugConf=xyz789...

Routes to Developer C's registered client connection and laptop.

No temporary per-developer ingress is needed. The static canary selects the shared router, while Redis performs the fast, dynamic session-to-developer decision.

Operational behavior

No sprLocalConnect cookieThe request remains on the normal QA6 WebUI backend.
Valid tunnel cookiesIngress selects the router; Redis resolves the session; the request reaches the developer laptop.
Old SPR_STICKINESSThe request may remain pinned to normal QA6. Delete the cookie before enabling local routing.
Missing session in HTTP WebUI modeThe router validates the original QA6 host, rebuilds its HTTPS URL, removes the LocalConnect routing cookies, and replays the HTTP request to normal WebUI. If validation or replay fails, it returns 502 with {"error":"tunnel-not-found"}.
Missing session in CONNECT modeThe proxy returns 502 Tunnel Not Found to the gRPC client. Raw CONNECT traffic cannot be replayed through the original WebUI ingress.
Relay restartThe client uses the coordinator on 8090 to request reassignment and reconnect.
Local WebUI stoppedThe request reaches the laptop but local forwarding returns an error.
Tunnel script stoppedPorts 4040, 8090 and 2222 close; Redis session state expires by TTL.
